## 1. 现状与问题

V0.1 现状（已上线）：

- 背景 `board_map.png`（1920×1080）**本身已画了一条 S 形取经路**，串起花果山→…→西天灵山八区域。
- 但 72 格用 `BoardManager._compute_position()` 的**蛇形 8×9 网格**硬排（`ORIGIN + col*120 / row*120`），**完全无视背景里的路**。
- 背景被压到 45% 透明度当陪衬，格子是彩色/图标圆——视觉上「棋盘浮在画上」，两者割裂。
- 窗口固定 1280×720，非全屏。

**根因**：格子坐标是程序生成的规则网格，和美术路线是两套独立坐标系，没有对齐。

## 3. 方案选型：72 格坐标如何贴合路线

这是核心决策。三条路线，给出推荐。

### 方案 A：美术重画「带 72 格点位」的新背景 + 代码读取点位表 ✅ 推荐

**做法**
1. **Codex 重画一张背景**：在现有取经路基础上，**沿路均匀画出 72 个格位凹槽/光点**（像大富翁/飞行棋那样路面嵌格子），保证视觉上 72 个落点清晰、间距均匀、首格花果山、末格灵山。
2. Codex 同时**交付一张坐标表** `data/board_path.json`：72 个点的归一化坐标（0–1，相对底图宽高），由 Codex 在生成时按实际画的格位标注。
3. **代码改 `BoardManager`**：弃用蛇形网格，改为读 `board_path.json` 的归一化点位 × 当前背景显示尺寸 → 实际屏幕坐标。格子图标可弱化甚至只留高亮（路面已画好坑位）。

**优点**
- 视觉最佳：格位是美术「画进」路面的，天然融合（G1+G2 一步到位）。
- 代码简单：只是把「算网格」换成「查表 × 缩放」。
- 数据驱动延续：点位进 `data/`，符合现有约定。

**缺点 / 成本**
- 依赖 Codex 重画 + 标点位（美术线一次交付）。
- 点位表需与背景图**严格对应**（改图必须同步改表）——靠命名契约约束。

### 方案 B：代码沿一条「贝塞尔/折线路径」程序化均匀采样 72 点

**做法**：在代码里手工定义一条与背景路线大致吻合的曲线（Path2D / 多段贝塞尔），用 `Curve2D.sample_baked()` 等距取 72 点。背景沿用现图。

**优点**：不依赖美术重画，纯代码可调；曲线可微调到贴路。
**缺点**：要**手工对着背景调曲线控制点**，反复试错且和这张特定背景强耦合；背景路是手绘不规则的，曲线很难处处贴合，仍可能「点不在路正中」。本质是用代码逼近美术，事倍功半。

### 方案 C：美术只重画背景（更宽的清晰路），代码仍程序化采样

A 和 B 的折中。仍有 B 的「代码逼近美术」问题。不推荐。

### 选型结论

**推荐方案 A**。它把「点位」这件事交给最擅长的一方（美术直接画在路上并给坐标），代码只做确定性的「查表×缩放」，融合度和可维护性都最好，且完全延续项目「数据驱动 + 美术/编码分工」的既定原则。

> 若希望**先不等美术、先把全屏和缩放框架搭好**：可先用方案 B 临时手调一条路径占位（标注 TODO），待 Codex 交付 A 的背景+点位表后替换。是否要这个过渡，见 §7 讨论点。

## 4. 方案 A 的具体改动

### 4.1 美术线（Codex + imagegen）交付物

| 交付 | 说明 |
|---|---|
| `assets/backgrounds/board_map.png`（重画） | 16:9，沿取经路均匀嵌入 72 个清晰格位（凹槽/莲台/石板），首=花果山、末=灵山；八区域视觉分段保留 |
| `data/board_path.json` | 72 点归一化坐标 `[{ "index":1, "x":0.07, "y":0.82 }, …]`，x/y ∈ [0,1] 相对底图。**由 Codex 按实际画的格位标注** |

> 坐标系约定：x 向右、y 向下，原点左上，归一化到底图宽高（与渲染缩放无关）。

### 4.2 编码线（Claude Code）改动

| 文件 | 改动 |
|---|---|
| `scripts/BoardManager.gd` | 新增加载 `board_path.json`；`get_tile_position(i)` 改为 `归一化点 × 背景显示矩形`；删除/降级蛇形 `_compute_position`（保留为 fallback：点位表缺失时仍能跑） |
| `scripts/Board.gd` / `Tile.gd` | 格子渲染弱化（路面已画坑位）：可只画高亮描边或小标记，棋子直接落点位；保留调试编号 |
| `scenes/GameScene.tscn` | 背景铺满并作为坐标基准；棋盘/棋子层依据背景显示矩形定位；背景透明度调回 ~100% |
| `project.godot` / 启动 | 全屏 + 自适应缩放（见 §5） |
| `data/board_path.json` | 新增（Codex 产出，编码线只读） |

### 4.3 坐标换算核心

```text
设背景显示区域为矩形 R（含全屏缩放后的实际位置/尺寸）。
tile_world_pos(i) = R.position + Vector2(path[i].x * R.size.x, path[i].y * R.size.y)
```

棋子同格四角偏移、移动动画沿 `get_move_path()` 逐点过——**这些逻辑都不用动**，只是底层坐标来源从「网格」换成「点位表×缩放」。

## 5. 全屏与自适应（G3）

macOS 尽量全屏 + 不变形，做两件事：

1. **启动全屏**：`project.godot` 设 `display/window/size/mode = 3`（Fullscreen）或 `4`（Exclusive Fullscreen）；或代码 `_ready` 里 `DisplayServer.window_set_mode(WINDOW_MODE_FULLSCREEN)`。提供 `Esc`/`F11` 退出/切换。
2. **自适应缩放**：沿用现有 `stretch_mode = "canvas_items"` + `aspect = "keep"`，把**设计分辨率提到 1920×1080**（与底图同比），全屏后等比缩放、留黑边但不变形。背景按设计分辨率铺满，点位×缩放自然跟随。

> 取舍：`keep`（保比例、可能黑边）vs `expand`（铺满、可能露出超出区域）。推荐 `keep`，配 16:9 底图在大多数 Mac（16:10/16:9）上黑边很小。讨论点见 §7。

## 8. 风险

- **背景与点位表不同步**：改图未改表会导致格子飘移。对策：成对交付 + 命名契约 + 一个校验脚本（点位数==72、范围 0–1）。
- **全屏不同 Mac 分辨率**：用 `keep` + 设计分辨率可控，已规避变形。
- **回归**：坐标来源改变可能影响碰撞/同格渲染——以「72 格数据契约不变、仅坐标层替换」隔离风险，跑一遍既有 1000 局/实机回归即可。
